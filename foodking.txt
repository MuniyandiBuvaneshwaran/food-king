1.1 SYSTEM OVERVIEW
--------------------
FoodKing is a full-stack food ordering and delivery platform. It covers:
- A RESTful JSON API backend (Laravel/PHP)
- An admin dashboard (Vue.js 3 SPA)
- A customer-facing website (Vue.js 3 SPA)
- A customer mobile app (Flutter/Dart)
- A delivery boy mobile app (Flutter/Dart)

The system supports multi-branch restaurant operations with online ordering,
dine-in table QR ordering, POS terminal ordering, kiosk ordering,
delivery management, kitchen display, and comprehensive reporting.

1.2 SYSTEM SCOPE
-----------------
- Multi-channel ordering: website, mobile app, POS, dining table QR, kiosk
- Multi-branch support with semi-independent operations per branch
- Complete order lifecycle (pending → confirmed → preparing → prepared → delivering → delivered)
- 22+ payment gateways with regional support
- Kitchen Display System (KDS) for kitchen staff
- Order Status Screen (OSS) for in-store customers
- Delivery boy assignment and GPS tracking
- Dining table QR code ordering
- Self-service kiosk terminals
- Sales, item, and credit balance reports
- Real-time notifications via Email, SMS, Push (Firebase)
- E-Wallet / Credit System for customers

2.2 AUTHENTICATION FLOWS
--------------------------
System uses token-based authentication (Laravel Sanctum).
Token included in all requests via Authorization: Bearer <token>
Token has no built-in expiration but can be revoked via logout.

2.2.1 Standard Login:
  1. Client sends email/phone + password
  2. Server validates credentials
  3. Returns personal access token + user profile

2.2.2 OTP-Based Signup (3-step):
  Step 1 — Request OTP: send email/phone → server generates OTP (4/6/8 digits),
            stores with expiry (5/10/15/20/30/60 min), delivers via SMS/email/both
  Step 2 — Verify OTP: send OTP code → server validates, returns success
  Step 3 — Complete Registration: send full details → server creates Customer user + token

2.2.3 Guest Signup:
  - Simplified path, no OTP required
  - Creates Customer account (name, email, phone, password)
  - Used when guest checkout converts to account

2.2.4 Social Login (Google):
  1. Client gets Google access token
  2. Sends to social login endpoint
  3. Server validates with Google, retrieves email/profile
  4. If user exists: returns token; if not: creates Customer + token
  5. Config (client ID, secret) stored in gateway_options table

2.2.5 Kiosk Machine Login:
  - Specialized login for self-service kiosk terminals
  - Authenticates with assigned credentials, receives branch-scoped token

2.2.6 Forgot Password (3-step):
  Step 1: Send email/phone → OTP generated and sent
  Step 2: Verify OTP
  Step 3: Send new password → server updates password

2.2.7 Token Refresh:
  - Endpoint revokes current token and issues a new one

SECTION 5: STATE MACHINES
================================================================================

5.1 ORDER STATE MACHINE
------------------------
Pending (5)
  → Confirmed (10)   [restaurant accepts]
  → Rejected (40)    [restaurant rejects]
  → Canceled (35)    [customer cancels]

Confirmed (10)
  → Preparing (15)   [kitchen starts]
  → Canceled (35)

Preparing (15)
  → Prepared (20)    [kitchen finishes]

Prepared (20)
  → Out for Delivery (25) [delivery boy picks up — for DELIVERY type]
  → Delivered (30)        [for TAKEAWAY or POS or TABLE type]

Out for Delivery (25)
  → Delivered (30)        [delivery boy marks delivered]

Delivered (30)
  → Returned (45)         [return initiated]

Any state → Canceled (35) by admin/branch manager at appropriate stages

5.2 PAYMENT STATE MACHINE
--------------------------
UNPAID (5) → PAID (10) [payment confirmed]

5.3 ORDER TYPE FLOW DIFFERENCES
---------------------------------
DELIVERY (5):    Customer address required, delivery boy assigned, full lifecycle
TAKEAWAY (10):   Customer picks up, no delivery boy, skips out-for-delivery state
POS (15):        Placed at POS terminal by operator, immediate payment
TABLE (20):      QR code scan, dine-in, no delivery, assigned to dining table

DELIVERY CHARGE CALCULATION:
  distance = Haversine formula between branch lat/lng and customer address lat/lng
  if distance ≤ order_setup_free_delivery_kilometer → delivery_charge = 0
  else → delivery_charge = basic_charge + (charge_per_kilo × (distance − free_km))
  Example: free=2km, basic=$1, per_km=$1, distance=5km → charge = $1 + ($1×3) = $4

SECTION 10: EDGE CASES & BUSINESS RULES
================================================================================

1. DELIVERY CHARGE:
   - Distance ≤ free_delivery_km → charge = 0
   - Distance > free_delivery_km → charge = basic_charge + (per_km_charge × extra_km)

2. COUPON APPLICATION:
   - Validate: active, within date range, usage limit not exceeded, min order met
   - PERCENTAGE type: capped by maximum_discount_amount if set
   - Applied before tax calculation on subtotal

3. E-WALLET / CREDIT:
   - Customer credit_balance field on users table
   - Credits applied at checkout (reduces amount due)
   - Credits issued on refund/cashback by admin

4. GUEST CHECKOUT:
   - guest login enabled via site_guest_login setting
   - Guest creates temporary account without OTP
   - Can be converted to full account post-order

5. SCHEDULED ORDERS:
   - Must select a date and time slot from available slots
   - Slot availability based on branch time slots
   - Slot duration configurable (default 30 minutes)

6. SOFT DELETES:
   - Users, Orders, Items, Branches etc. all soft-deleted (deleted_at timestamp)
   - Deleted records excluded from normal queries

7. IMAGE CONVERSIONS (auto-generated at upload via Spatie):
   - thumb: 168×180 px
   - cover: 390×270 px
   - preview: 600×600 px
   - All crop-to-fit, original preserved

8. ORDER SERIAL NUMBER:
   - Format: DDMMYY + ORDER_ID
   - Example: March 25 2025, ID 42 → "25032542"

9. BRANCH SCOPE:
   - BranchScope global query scope auto-applied
   - Branch Manager can only see data for their branch
   - Admin can see all branches
